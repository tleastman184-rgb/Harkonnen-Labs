## Phase B — Trace Spine

**Closes gap:** #3 (structured, queryable reasoning traces) and #8 (auditability).

**Why first:** Every other improvement loop (C, D) depends on having structured per-agent
reasoning traces. Without traces, Coobie pattern-matches over text summaries. With traces,
she can answer "which reasoning step preceded all failures of type X?"

### What to build

- `AgentTrace` struct: `trace_id`, `run_id`, `agent`, `phase`, `input_summary`,
  `reasoning_steps: Vec<String>`, `actions_taken: Vec<String>`, `outcome`,
  `input_tokens`, `output_tokens`, `latency_ms`, `created_at`
- `agent_traces` SQLite table — append-only, one row per significant agent LLM call
- Reasoning steps extracted from LLM responses: agents that emit structured output
  prepend a short `<reasoning>` block; this is parsed and stored separately from the
  final output content
- `GET /api/runs/:id/traces` endpoint returning `Vec<AgentTrace>`
- Wire at: Scout intake, Mason plan generation, Mason edit generation, Coobie critique,
  Coobie briefing, Sable scenario evaluation

### Files to change

- `src/models.rs` — `AgentTrace` struct
- `src/db.rs` — `agent_traces` table + index on `(run_id, agent, created_at)`
- `src/orchestrator.rs` — `record_agent_trace` helper; call sites at each agent LLM call
- `src/api.rs` — `GET /api/runs/:id/traces` route + handler

### What shipped

- `AgentTrace` struct in `src/models.rs`
- `agent_traces` table + index on `(run_id, agent, created_at)` in `src/db.rs`
- `record_agent_trace` + `list_run_traces` helpers in `src/orchestrator.rs`
- `extract_reasoning()` + `trace_input_summary()` free functions
- Wired at: Scout intake, Coobie briefing, Coobie critique, Mason plan, Mason edits, Sable (orchestrator level)
- `GET /api/runs/:id/traces` route + handler in `src/api.rs`
- Build clean (0 errors, 6 warnings — all pre-existing)

## Phase C — OptimizationProgram

**Closes gap:** #6 (explicit intent → plan separation) and #1 (program-level objectives
that the authority layer can enforce).

**Why:** Most current specs do not have a machine-readable success metric. This makes
the plan critique soft — Coobie can flag guardrails but cannot check "does this plan
actually optimize the stated objective?" Adding an `OptimizationProgram` per run gives
Scout, Bramble, and Coobie a shared, explicit definition of success.

### What to build

- `OptimizationProgram` struct: `objective_metric`, `objective_description`,
  `editable_surface: Vec<String>`, `constraints: Vec<String>`,
  `evaluation_plan: Vec<String>`, `time_budget_secs: Option<u64>`
- Generated by Scout during intake as a second output alongside `IntentPackage`
- Written to `optimization_program.json` in the run workspace
- Coobie critique updated: check whether the plan addresses the stated objective metric
- Bramble (when real test execution ships in Phase 2) will validate that test commands
  measure the objective
- `GET /api/runs/:id/optimization-program` endpoint

### Files to change

- `src/models.rs` — `OptimizationProgram` struct
- `src/orchestrator.rs` — `scout_derive_optimization_program` helper called after
  `scout_intake`; written to run workspace; passed into `coobie_critique_plan`
- `src/api.rs` — `GET /api/runs/:id/optimization-program` route + handler

### What shipped

- `OptimizationProgram` struct in `src/models.rs`
- `scout_derive_optimization_program` helper in `src/orchestrator.rs` — LLM-backed with stub fallback
- Called after `scout_intake` in the run flow; writes `optimization_program.json` to workspace
- `coobie_critique_plan` extended: accepts `Option<&OptimizationProgram>`, adds advisory concern when plan doesn't address objective metric
- `GET /api/runs/:id/optimization-program` route + handler in `src/api.rs`
- Build clean (0 errors)

## Phase D — Adversarial Sable (MetricAttack)

**Closes gap:** #2 (world model) partially and #3 (closed-loop verification) partially.

**Why:** Sable already runs hidden scenarios. The upgrade is making her actively try to
*break* the evaluation metric rather than just probe edge cases. A system that can generate
its own red-team attacks against its own metrics is structurally much harder to game.

### What to build

- `MetricAttack` struct: `attack_id`, `run_id`, `exploit_description`,
  `detection_signals: Vec<String>`, `mitigations: Vec<String>`, `status`
- Sable's scenario generation adds an attack-generation step: given `OptimizationProgram.objective_metric`,
  generate 2–3 ways an implementation could cheat it (shallow tests, output caching, hardcoded
  return values, etc.)
- Each attack becomes a probe scenario appended to the hidden scenario bundle
- Results written to `metric_attacks.json` in the run workspace alongside the scenario results
- `metric_attack_detected: bool` flag on the run summary
- `GET /api/runs/:id/metric-attacks` endpoint
- Coobie consolidation: when an attack succeeds, generate a lesson
  "spec X's metric was gameable via Y — add Z constraint to evaluation_plan"

### Files to change

- `src/models.rs` — `MetricAttack` struct
- `src/orchestrator.rs` — `sable_generate_metric_attacks` called after scenario generation
  if an `OptimizationProgram` exists for the run; `record_metric_attacks` writes to workspace
- `src/api.rs` — `GET /api/runs/:id/metric-attacks` route + handler

### What shipped

- `MetricAttack` struct in `src/models.rs`
- `sable_generate_metric_attacks` helper in `src/orchestrator.rs` — LLM-backed, generates 2-3 attacks per run
- Called immediately after `scout_derive_optimization_program` in the run flow; writes `metric_attacks.json`
- Each attack: exploit description, detection signals, suggested mitigations
- Agent trace + cost event recorded for Sable's attack generation LLM call
- `GET /api/runs/:id/metric-attacks` route + handler in `src/api.rs`
- Build clean (0 errors)

### Status: ✅ SHIPPED

## Phase E — Outcome Observation Loop (DEFERRED)

**Closes gap:** #2 (world state), #3 (outcome verification), #7 (external interfaces).

**Why deferred:** Requires a live system state graph for meaningful drift detection. The
TypeDB layer (ROADMAP Phase 6) provides that backing. Without it, observations are stored
but Coobie cannot reason over them in a typed, queryable way.

### What will be built (when unblocked)

- `ObservationHook` model — post-run checks that fire after deployment/artifact delivery
- `POST /api/runs/:id/observe` — accepts `{ metric, value, observed_at }` payloads
- `run_observations` table — drift events stored with configurable threshold
- `drift_detected` live event emitted when observed diverges from expected
- Coobie ingests drift events as high-signal causal evidence into next run's preflight
- First external integration point: simple CI/CD webhook + Prometheus scrape endpoint

### Unblocked by: ROADMAP Phase 6 (TypeDB)
